## 4. RÈGLES CRITIQUES (à respecter absolument)

### Architecture

1. **Structure cible** :
   - `shared/ui/` = composants UI atomiques (réutilisables partout)
   - `shared/utils/` = fonctions pures sans state
   - `shared/services/` = wrappers API
   - `shared/state/` = état global volatile (tabState)
   - `features/<domaine>/screens/` = écrans métier
   - `features/<domaine>/context/` = contexts par domaine (à créer)
   - `features/<domaine>/modals/` = modales métier (à créer)

2. **Ne PAS remettre des fichiers dans `components/screens/` plat** — c'était le bug d'architecture de Phase 4 corrigé par Phase 5.

3. **Les screens reçoivent leurs closure refs EN PROPS** jusqu'à Phase 7. Après Phase 7, via context.

### Règles de refactor

4. **Une extraction = un patch = un commit = un deploy** (jamais chaîner plusieurs phases en un push)
5. **Backup auto avant modification** — le patcher Phase N crée toujours `App.jsx.pre-phaseN-<ts>`
6. **Rollback prêt avant chaque deploy** — `rollback-phaseN.sh` pour chaque phase
7. **Si extraction nécessite > 10 props → stop**, créer un context d'abord
8. **Si un composant a > 1 500 lignes après extraction → subdiviser**

### Règles de build

9. **Jamais `--minify=false`** sur le VPS (OOM 3+GB RAM)
10. **Toujours copier dist → httpdocs** après `npm run build` (sinon nginx sert l'ancienne version)
11. **Vérifier HTTP 200 après deploy** (smoke-test minimal)

### Backend (§0 de CLAUDE.md — règle absolue)

12. **Jamais `import { db } from '../db/database.js'` direct** dans les routes backend. Toujours passer par `tenantResolver`.
13. **Chaque nouvelle entreprise = sa propre DB SQLite** dans `/var/www/planora-data/tenants/`
