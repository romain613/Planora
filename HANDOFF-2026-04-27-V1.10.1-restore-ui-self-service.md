## 1. Contexte

V1.10.0 avait livré le restore engine backend (R1+R2). V1.10.1 livre l'UI self-service collab + un fix critique découvert au test :

**🚨 Bug racine identifié au test V1.10.0** : `scope.js` filtrait contacts via `ownerCollaboratorId OR executorCollaboratorId` — colonnes existantes mais **vides en prod**. Le système utilise réellement `assignedTo` (95% des cas), `sharedWithId` (Contact Share V1), `shared_with_json` (cross-collab V1.8.13).

Conséquence avant fix : **les 49 snapshots prod existants avaient contacts=0**. Restaurer un de ces snapshots **aurait supprimé tous les contacts du collab**. Garde-fou anti-perte ajouté en parallèle.

## 2. Patches livrés

### A — Frontend UI self-service (CollabPortal.jsx)

**Sub-tab Settings "Sauvegardes & restauration"** (icône 🛡 vert) :
- Header rassurant : "Vos données sont sauvegardées automatiquement" + "Dernière sauvegarde : il y a X min"
- Bouton "Créer une sauvegarde maintenant" (POST /manual)
- Banner "Annuler la restauration" affiché 1× après chaque restore (préparation R4 livrée)
- Liste des sauvegardes avec format humain ("il y a 2 min", "aujourd'hui 09:15", "hier 14:30", "27 avr 14:30")
- Tags visuels : 🔄 Auto / 👆 Manuel / ⏪ Pré-restauration
- 3 boutons par ligne : **Voir les détails** / **Restaurer cet état** / **🗑 Supprimer**
- Bandeau info bleu : "Conservation : les sauvegardes de plus de 7 jours sont automatiquement supprimées"

**Modal "Voir les détails"** (GET /:id/preview) :
- 6 cartes counts visuels : Contacts / **RDV enregistrés** / Suivis pipeline / Changements statut / Notes IA / Documents
- Bloc "Vos 3 prochains RDV" (sample depuis state local `bookings` pour aperçu concret de l'état actuel qui sera remplacé)
- Warnings séparés : critical (rouge bordure 2px, bouton restore désactivé) vs simple (jaune)
- CTA "Restaurer cet état" (désactivé si critical)

**Modal de confirmation Restore** (obligatoire) :
- Titre "Revenir à cet état ?"
- Bandeau ⚠ "Vos données actuelles seront remplacées"
- Mention 🛡 "Une sauvegarde sera créée. Vous pourrez annuler cette action."
- Bouton "Confirmer la restauration" (loader pendant traitement, désactivation Annuler)

**Sync UI sans reload (critique)** post-restore :
1. Reset states locaux : `selectedCrmContact`, `pipelineRightContact`, `phoneActiveCall`, `cockpitOpen`, `cockpitMinimized`
2. Refetch global : `Promise.all([api('/api/data/contacts'), api('/api/bookings')])`
3. **Event global** : `window.dispatchEvent(new Event('collabDataRestored'))`
4. **useEffect listener** : refetch automatique pour tous les modules
5. Refresh liste sauvegardes
6. Toast `✅ Vos données ont été restaurées`
7. **AUCUN `window.location.reload()`** ✅

### B — Backend fix scope contacts (CRITIQUE)

**`scope.js`** ([server/services/collabSnapshots/scope.js](server/services/collabSnapshots/scope.js))
- Filtre contacts élargi à 5 colonnes ownership :
  ```sql
  WHERE companyId = ? AND (
    ownerCollaboratorId = ?
    OR executorCollaboratorId = ?
    OR assignedTo = ?
    OR sharedWithId = ?
    OR shared_with_json LIKE '%"' || ? || '"%'
  )
  ```
- Match strict `'%"<id>"%'` pour shared_with_json (matche `["u-jordan"]` mais pas `"u-jordan-truc"`)

**`fingerprint.js`** ([server/services/collabSnapshots/fingerprint.js](server/services/collabSnapshots/fingerprint.js))
- Aligné sur le même filtre. Sans cet alignement, les changements `pipeline_stage` sur contacts `assignedTo` ne déclenchaient pas de snapshot dirty.

### C — Garde-fou anti-perte (CRITIQUE)

**`previewRestore.js`** : warning `kind: 'critical-empty-snapshot'` quand `currentContacts > 5 && snapshotContacts === 0`.

**`restoreSnapshot.js`** : refus immédiat avec message explicite. Override possible via `reason="force-empty-snapshot-override"` (admin uniquement, CLI direct).

**Frontend** : warnings critical affichés en rouge bordure 2px avec icône `shield-off` "Restauration bloquée". Bouton "Restaurer cet état" désactivé + tooltip.

### D — Route DELETE backend (audit §5.2 livré)

**`routes/collabSnapshots.js`** : nouvelle route `DELETE /api/collab-snapshots/:id`
- Auth scoping : self ou admin (réutilise `canAccessCollab()`)
- **Garde-fou pre-restore non expirée** : retourne **HTTP 409** "protégée pour vous permettre d'annuler une restauration récente"
- DELETE row + suppression fichier .gz best-effort
- audit_logs immutable : `action: collab_snapshot_deleted`, `category: data-recovery`
