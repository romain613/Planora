## PHASE 2 — Fondations : git (local-first) + backup cron + smoke tests (Jour 3-5)

**Objectif** : sécuriser AVANT toute modif. Inclut smoke test pour chaque phase suivante.

### 2A — Git init DEPUIS LE MAC (pas le VPS) — ⚠ CHANGEMENT v1.1

**Rationale** : le code vit sur le repo git (source de vérité), pas sur le VPS. Le VPS devient une **cible de déploiement**. Si on `git init` sur le VPS, tout hotfix SSH direct diverge silencieusement du repo et on perd la traçabilité.

```bash
# Sur le Mac MH (pas SSH)
cd ~/Documents/planora-code

cat > .gitignore <<'EOF'
node_modules/
dist/
build/
*.log
*.db
*.db-wal
*.db-shm
.env
.DS_Store
App.jsx.bak*
App.jsx.pre-*
database.js.bak*
transfer.js.bak*
*.bak-*
/server/.env
EOF

git init -b main
git add .
git commit -m "Initial commit - etat prod 2026-04-19 apres V7 P0"
# Créer le repo privé sur GitHub (jamais public)
git remote add origin git@github.com:<org>/planora.git
git push -u origin main
```

**Corollaire** : ajouter dans `deploy-all.sh v19` un check qui refuse de déployer si le working tree VPS diverge du tag git attendu (protège contre les hotfix SSH orphelins).

### 2B — Backups VPS automatisés (cron quotidien)
```bash
ssh planora-vps "cat > /usr/local/bin/planora-backup.sh <<'EOF'
#!/bin/bash
STAMP=\$(date +%Y%m%d-%H%M%S)
BACKUP_DIR=/var/backups/planora/full-\$STAMP
mkdir -p \"\$BACKUP_DIR\"/{dbs,code,configs}
sqlite3 /var/www/planora-data/calendar360.db \".backup \$BACKUP_DIR/dbs/calendar360.db\"
sqlite3 /var/www/planora-data/control_tower.db \".backup \$BACKUP_DIR/dbs/control_tower.db\"
for db in /var/www/planora-data/tenants/*.db; do
  sqlite3 \"\$db\" \".backup \$BACKUP_DIR/dbs/tenant-\$(basename \$db)\"
done
tar --exclude='node_modules' --exclude='*.log' -czf \"\$BACKUP_DIR/code/server.tar.gz\" -C /var/www/planora server
tar czf \"/var/backups/planora/planora-backup-\$STAMP.tar.gz\" -C /var/backups/planora \"full-\$STAMP\"
chmod 600 \"/var/backups/planora/planora-backup-\$STAMP.tar.gz\"
ls -1t /var/backups/planora/planora-backup-*.tar.gz | tail -n +8 | xargs rm -f
ls -1t /var/backups/planora/full-* -d | tail -n +8 | xargs rm -rf
EOF
chmod +x /usr/local/bin/planora-backup.sh
(crontab -l 2>/dev/null; echo '0 3 * * * /usr/local/bin/planora-backup.sh >> /var/log/planora-backup.log 2>&1') | crontab -"
```

Validation : lancer le script une fois manuellement, vérifier que le tarball fait 25-50 Mo.

## PHASE 5 — Migration progressive routes → tenantResolver (Semaines 3-5)

**Objectif** : éliminer tous les `import { db }` des routes backend (respect §0).

### 5A — Inventaire + priorisation (issu audit 1B)
| Criticité | Exemples | Effort/route | Risque |
|---|---|---|---|
| **Haute** | `voip.js`, `contacts.js`, `bookings.js`, `ai-copilot.js` | 1-2h | Élevé (prod live CapFinances+MonBilan) |
| **Moyenne** | `messaging.js`, `pipeline.js`, `forms.js` | 1h | Moyen |
| **Basse** | `settings.js`, `templates.js`, routes admin | 30min | Faible |

### 5B — Pattern de migration
**Avant** :
```js
import { db } from '../db/database.js';
router.get('/endpoint', (req, res) => {
  const rows = db.prepare('SELECT * FROM contacts WHERE companyId = ?').all(req.auth.companyId);
});
```

**Après** :
```js
import { getDbForCompany } from '../db/tenantResolver.js';
router.get('/endpoint', (req, res) => {
  const db = getDbForCompany(req.auth.companyId, 'contacts');
  const rows = db.prepare('SELECT * FROM contacts WHERE companyId = ?').all(req.auth.companyId);
});
```

### 5C — Helper `getDbForCompany` — VERSION CORRIGÉE v1.1

**⚠ CHANGEMENT v1.1** : la version naïve (`case 'shadow': return db`) PERD le shadow-compare. Version correcte :

```js
// Dans tenantResolver.js
export function getDbForCompany(companyId, feature) {
  const mode = getRouteMode(companyId, feature);
  switch (mode) {
    case 'tenant':
      return getTenantDb(companyId);
    case 'shadow':
      // Lecture monolithe MAIS shadow-compare async en background
      scheduleShadowCompare(companyId, feature);
      return db;
    case 'legacy':
    default:
      return db;
  }
}

// Helper explicite si l'appelant a besoin des 2 handles simultanément
export function getDualDbForCompany(companyId, feature) {
  return {
    primary: db,
    shadow: getTenantDb(companyId),
    mode: getRouteMode(companyId, feature)
  };
}
```

Les routes qui font des lectures critiques (ex: audit comparison) utilisent `getDualDbForCompany`. Les routes normales utilisent `getDbForCompany`.

### 5D — Ordre d'attaque (1 route = 1 PR = 1 deploy + observation 24h)
1. Routes basses d'abord (settings.js, templates.js) → calibrage du pattern
2. Puis routes moyennes (pipeline.js, forms.js)
3. Enfin routes hautes (voip.js, contacts.js, bookings.js, ai-copilot.js) — **1 seule à la fois**, 48h d'observation entre chaque

## PHASE 7 — Migration companies legacy → tenant (Semaines 12-20)

**Objectif** : passer les 4 entreprises restantes du mode `legacy` au mode `tenant`, via `shadow`. Monolithe devient archive.

**⚠ CLARIFICATION v1.1** : le timing dépend du parallélisme choisi :
- **Serial** (safe) : 4 companies × 2 semaines shadow chacune = **8 semaines minimum**
- **Parallèle** (risque) : toutes en shadow en même temps = 2-3 semaines, mais si bug sur 1 → impact sur les 4

**Reco** : **mi-parallèle** — grouper par 2, shadow 2 semaines, bascule, puis 2 suivantes. **Timing réaliste : 6-8 semaines**.

### 7A — Provisioning tenant DBs manquantes
Pour chaque company legacy :
```js
// Pseudo-code
1. Créer /var/www/planora-data/tenants/<companyId>.db
2. Initialiser schéma complet (93 tables via sync-tenant-schemas.js)
3. Enregistrer dans control_tower.tenant_databases
4. Copier données monolithe WHERE companyId = ?
5. UPDATE companies SET tenantMode='shadow', tenantFeatures='{"contacts":"shadow"}'
6. invalidateTenant(companyId)
```

### 7B — Observation shadow (2 semaines par company/groupe)
Surveiller `tenant_shadow_diffs`. Si divergences → fixer avant bascule.

### 7C — Bascule mode tenant
Après 7 jours consécutifs sans divergence :
```sql
UPDATE companies SET tenantMode = 'tenant' WHERE id = ?;
```
Monolithe devient read-only pour cette company.

### Critères fin de phase
- ✅ 6/6 companies en mode tenant
- ✅ Monolithe sans écriture depuis 30 jours
- ✅ Possibilité de purger les rows monolithe par company (gain place)

### Rollback
- Divergence shadow inattendue → `UPDATE companies SET tenantMode='legacy'` → retour monolithe
- Bascule tenant casse → rollback via `tenantMode='shadow'` + investigate
